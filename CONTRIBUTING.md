## Working with Data Files

### Directory Structure

```
config/            ← User-editable configurations (committed to git)
  ├── physical.json
  ├── traffic.json
  ├── vlan.json
  └── settings.json

public/data/       ← Runtime data (gitignored, generated by watcher)
  └── *.json
```

### Data Files

The `config/` directory contains default network topology configurations committed to git:

- **`physical.json`** - Physical network topology with nodes and connections
- **`traffic.json`** - Traffic flow patterns between nodes
- **`vlan.json`** - VLAN configurations and assignments
- **`settings.json`** - Application settings

These defaults are baked into the Docker image so the UI works out of the box. Override them by mounting a local directory at `/app/config` when running the container.

### Validation

During development, the watcher automatically validates data files:

```bash
# Development mode (runs vite + watcher)
pnpm dev

# Or run watcher standalone
pnpm watch
```

See `schema/` directory for JSON schemas that define the expected structure.

### How Validation Works (Zero Downtime)

When you modify a JSON data file, the watcher validates its content against the required schemas.

- **If valid**: The new data is instantly served to the application.
- **If invalid**: The application continues to use the _last valid_ version of your data to prevent crashes. The watcher generates a `.meta.json` file (e.g., `physical.meta.json`) containing the validation errors (`valid: false`, `validationErrors: [...]`). The React app fetches this metadata and displays warning banners in the UI, indicating exactly what failed in your file.

## Naming Conventions

- Boolean variables must use `is...` naming.
- Examples: `isOpen`, `isLoading`, `isLegendVisible`, `isNodeLabelsVisible`.
- Apply this to local variables, React state/ref booleans, boolean params, and boolean action payload fields.
- For external contracts (browser/library APIs, third-party schemas, or existing JSON field names), keep the required external key names and map them to local `is...` variables when needed.
- Any utility or helper function that can throw — either by containing an explicit `throw` statement, or by propagating an unhandled throw from a called `...OrThrow` function — must include `OrThrow` in its name.
- Examples: `getNodeTypeOrThrow` (direct throw), `fetchJsonOrThrow` (propagates `fetch()` network errors), `buildPageDataOrThrow` (propagates via `styleNodeOrThrow` / `styleEdgeOrThrow`).
- This convention does not apply to React components or to public-API hooks that throw only as a provider-missing safety check.
- Prefer moving inline `throw` logic into a named `...OrThrow` helper so throw behavior is explicit and discoverable.

## CSS File Conventions

This project uses two CSS approaches depending on whether styles need scoping.

### CSS Modules (Scoped Styles)

Use CSS Modules for component-specific styles where you control the `className`.

**File naming**: `ComponentName.module.css` (PascalCase, matching component name)

```
src/components/
└── MyComponent/
    ├── MyComponent.tsx
    └── MyComponent.module.css
```

**Class naming**: `camelCase` (accessed via JS object)

```css
/* MyComponent.module.css */
.container { ... }
.headerTitle { ... }
.actionButton { ... }
```

```tsx
import styles from "./MyComponent.module.css";
<div className={styles.container}>
  <h1 className={styles.headerTitle}>...</h1>
</div>;
```

### Global CSS (Unscoped Styles)

Use global CSS for:

- **Base styles**: Resets, variables, fonts, layout (`src/styles/base/`)
- **Third-party library overrides**: vis-network generates `.vis-tooltip`, Konva elements, etc.
- **Cross-component positioning**: When parent components need to position children (e.g., `.notification-stack`)

**File naming**: `kebab-case.css`

**Class naming**: `kebab-case`

```css
/* GraphView.css - must be global because vis-network generates these classes */
.vis-tooltip { ... }
.vis-tt-head { ... }
```

### Directory Structure

```
src/
├── styles/
│   ├── base/           ← Global base styles (variables, reset, fonts)
│   │   ├── variables.css
│   │   ├── reset.css
│   │   └── fonts.css
│   ├── components/     ← Global component utilities
│   │   └── page-loading.css
│   └── index.css       ← Main entry point (imports base styles)
└── components/
    └── MyComponent/
        ├── MyComponent.tsx
        └── MyComponent.module.css   ← Scoped to this component
```

### When to Use Which

| Scenario                            | Use         | Example                           |
| ----------------------------------- | ----------- | --------------------------------- |
| Component-specific styles           | CSS Modules | `AppHeader.module.css`            |
| Third-party library overrides       | Global CSS  | `.vis-tooltip` in `GraphView.css` |
| CSS variables, resets, fonts        | Global CSS  | `src/styles/base/variables.css`   |
| Styles accessed by string className | Global CSS  | Classes generated by libraries    |

## Code Documentation Standard

This project enforces a consistent documentation style across all file types.
The standard is designed so any contributor — human or AI — can apply it
without ambiguity. The guiding principle is: **document the _why_, not the _what_**.
Comments that restate what the code does are noise. Comments that explain
intent, constraints, and non-obvious decisions are signal.

### The Golden Rule

Before adding a comment, ask: "Would an experienced TypeScript/React developer
need this to understand the code?" If yes, write it. If no, skip it.

**Never add a comment for:**

- What a statement obviously does (`i++ // increment i`)
- Trivial getters whose name is already the documentation
- `color`, `margin`, `padding`, `display: flex`, or other universally understood CSS
- Standard Dockerfile instructions (`WORKDIR`, `EXPOSE`)
- Individual `export { X } from './X'` lines in barrel files
- Import lines

**Always add a comment for:**

- "Why not the obvious approach" reasoning
- Non-obvious algorithmic choices (hashing, monkey-patching, mutation tricks)
- Sentinel values and their invariants (`null` meaning "all flows")
- Order-sensitive operations
- Every `eslint-disable` directive — one-line reason on the same line
- Magic numbers that are not captured in a named constant
- Non-obvious CSS properties (`pointer-events: none`, unusual `z-index`, transform tricks)

### TypeScript and TypeScript React (`.ts` / `.tsx`)

Every source file opens with a JSDoc file header:

```typescript
/**
 * @file FileName.ts
 * @description One-sentence summary of what this file provides
 *
 * Optional: Design decisions, constraints, or "why not X" context.
 * Include only when the one-liner is insufficient.
 *
 * Exports:               ← list when 4+ things are exported
 * - FunctionName: what it does
 * - ConstantName: what it represents
 *
 * @example
 * // Non-obvious usage only
 * const result = myFunction(input);
 *
 * @see RelatedFile.ts - What it provides (dash + description required)
 */
```

For `.tsx` component files, add a component tree when the component has 3+ children or
conditional rendering:

```
 * Component Structure:
 *   ComponentName
 *   ├─ ChildA (description)
 *   └─ ChildB (description)
```

**Section dividers** — use when a file has 3 or more logical sections:

```typescript
/* ============================================================================
 * SECTION NAME
 * Optional one-line description (omit if name is self-explanatory)
 * ============================================================================ */
```

Width is exactly 80 characters. Do NOT add dividers to files with only 1–2 sections.

**Function JSDoc:**

```typescript
/**
 * One-sentence description of what this function does.
 *
 * Optional: context, "why not X" reasoning, or invariants.
 *
 * @param {Type} paramName - What this parameter controls
 * @param {Type} [optional=default] - Optional parameter description
 * @returns {Type} What is returned and when
 * @throws {Error} When and why this throws
 */
```

`@param` and `@returns` are **required** when:

- The function has more than one parameter
- A parameter's purpose is not obvious from its name
- The return value can be `null`, `undefined`, or has meaningful semantics

`@param` and `@returns` may be **omitted** for trivially self-documenting single-argument functions.

**Private helper functions** get a one-line JSDoc when their name alone is insufficient:

```typescript
/**
 * Build a human-readable label from a camelCase or snake_case key.
 * Uses SECTION_LABELS lookup first, then falls back to word-splitting.
 */
function sectionLabelFor(key: string): string { ... }
```

**Interface JSDoc** — add `@property` tags when properties have constraints or invariants:

```typescript
/**
 * Canonical page metadata used by tabs, routing, and tooltips.
 *
 * @property {PageId} id - Canonical route key used in URL hash
 * @property {string} label - Short tab label (≤ 20 chars)
 * @property {string} title - Descriptive subtitle shown in sidebar footer
 */
export interface Page {
  id: PageId;
  label: string;
  title: string;
}
```

**Inline comments** — use `/* */` (never `//`) inside function bodies:

```typescript
/* Split pathname into segments, filtering empty strings */
const segments = pathname.split("/").filter(Boolean);

/* Traffic page supports sub-page filtering — check second segment */
if (page === "traffic") return parseTrafficRoute(segments);
```

### Barrel Exports (`index.ts`)

Minimal form — use when the barrel re-exports a single component or runtime value:

```typescript
/**
 * @file index.ts
 * @description Barrel export for ComponentName
 *
 * Import style:
 * import { ComponentName } from '@/path/to/ComponentName';
 */
export { ComponentName } from "./ComponentName";
```

Detailed form — use when the barrel mixes types + runtime code, or exports 4+ things:

```typescript
/**
 * @file index.ts
 * @description Barrel export for [Module Name]
 *
 * Public API:
 * - FunctionA: what it does
 * - type TypeB: what it represents
 *
 * @see module.ts - Runtime helpers
 * @see module.types.ts - Type contracts
 *
 * Import style:
 * import { FunctionA, type TypeB } from '@/utils/module';
 */
```

### JSON Data Files (`config/*.json`)

A root-level `$comment` is required explaining what the file is, how to use it, and
which schema validates it:

```json
{
  "$comment": "Lab Map - Physical topology data. Defines network topology: nodes (devices), edges (cable connections), hardware specs, and switch port mappings. Validated by schema/physical.schema.json."
}
```

Individual node/edge entries do **not** receive `$comment` fields.

---

### JSON Schema Files (`schema/*.schema.json`)

Every top-level property and `definitions` entry requires both `$comment` (short one-liner)
and `description` (full human-readable explanation). Nested properties need `description` only.

---

### YAML, Dockerfile, Shell Scripts

```yaml
# =============================================================================
# @file filename.yml
# @description One-sentence purpose
#
# Explanation paragraph.
#
# Usage:
#   command to run
#
# Flow:
#   Step1 → Step2 → Step3
# =============================================================================
```

Dockerfile and shell scripts use the same `#` header format with `=` borders.
Shell scripts have `#!/bin/sh` (or `#!/bin/bash`) as the very first line before the header.
Dockerfile uses `# ---` dividers between build stages.

---

### HTML (`index.html`)

```html
<!--
  @file index.html
  @description One-sentence purpose

  Explain non-obvious sections (theme flash prevention, fallbacks, etc.)
-->
```

Inline `<!-- comments -->` for major sections only. Do not comment individual tags.

---

### Markdown (`.md`)

No file header block — H1 heading serves as the title. Every `.md` file in a
subdirectory (`schema/`, `docker/`, `data/`) must begin with an
`# H1` title followed by a one-sentence purpose statement. All code blocks must
have a language specifier (` ```bash `, ` ```json `, ` ```typescript `).

## Dependencies Overview

### Why Each Dependency Exists

**Runtime Dependencies** (required for app to function):

- **ajv** - Validates JSON topology files against schemas. Without it, invalid data could crash the app.
- **chokidar** - Watches data files for changes during development. Without it, you must manually revalidate after every edit.
- **react** - Core UI framework. App won't run without it.
- **react-dom** - Renders React to browser DOM. App won't display without it.
- **vis-network** - Renders interactive network graphs for physical/traffic/VLAN views. Without it, graph views won't work.

**Development Dependencies** (build tools, linters, type checkers):

- **@eslint/js** - Base ESLint JavaScript config
- **@types/react** - TypeScript types for React (enables type checking)
- **@types/react-dom** - TypeScript types for React DOM
- **@vitejs/plugin-react** - Enables React JSX + Fast Refresh in Vite
- **eslint** - Lints code for quality issues
- **eslint-import-resolver-typescript** - Resolves TypeScript imports (including `@/` aliases) for ESLint
- **eslint-plugin-import** - Enforces import order and organization
- **eslint-plugin-react** - React-specific linting rules
- **eslint-plugin-react-hooks** - Enforces React Hooks rules (prevents bugs like missing useEffect deps)
- **eslint-plugin-react-refresh** - Ensures Fast Refresh compatibility
- **typescript** - Compiles TypeScript and provides type checking
- **typescript-eslint** - TypeScript-specific ESLint rules
- **vite** - Dev server and build tool

Each package's usage location can be found by searching the codebase (e.g., `grep -r "vis-network" src/`).

## Creating Node Icons

When adding new device icons to `public/node-icons/`, follow these design guidelines.

### Directory Structure

```
public/node-icons/
└── {category}-{type}-{variant}/
    ├── dark.svg
    └── light.svg
```

**Naming convention**: `{category}-{type}-{variant}` where:

- **category**: `desktop`, `generic`, `server-rack-1u`, `server-rack-2u`
- **type**: `compute`, `router`, `switch`, `nas-hdd`, `nas-ssd`, `internet`, `vpn`
- **variant**: number starting at 1 (e.g., `desktop-compute-1`, `desktop-compute-2`)

### Valid Node Types

| Node Type                | Icon Folder                 | Category                |
| ------------------------ | --------------------------- | ----------------------- |
| `desktop-compute`        | `desktop-compute-1/`        | Desktop workstation     |
| `desktop-router`         | `desktop-router-1/`         | Desktop router/firewall |
| `generic-internet`       | `generic-internet-1/`       | Internet/WAN cloud      |
| `generic-vpn`            | `generic-vpn-1/`            | VPN tunnel              |
| `server-rack-1u-compute` | `server-rack-1u-compute-1/` | 1U compute server       |
| `server-rack-1u-nas-hdd` | `server-rack-1u-nas-hdd-1/` | 1U NAS (HDD)            |
| `server-rack-1u-router`  | `server-rack-1u-router-1/`  | 1U router/firewall      |
| `server-rack-1u-switch`  | `server-rack-1u-switch-1/`  | 1U managed switch       |
| `server-rack-2u-compute` | `server-rack-2u-compute-1/` | 2U compute server       |
| `server-rack-2u-nas-hdd` | `server-rack-2u-nas-hdd-1/` | 2U NAS (HDD)            |
| `server-rack-2u-nas-ssd` | `server-rack-2u-nas-ssd-1/` | 2U NAS (SSD)            |

### Design Requirements

1. **Flat colors only** - no gradients or complex fills
2. **Consistent stroke widths**:
   - `0.7` for fine details (ports, small elements)
   - `0.8` for panels and internal sections
   - `1.2-1.5` for main chassis outline
3. **Rounded corners**: use `rx="1"` for small elements, `rx="2"` or `rx="3"` for larger panels
4. **Simple shapes**: prefer rectangles, circles, and lines over complex paths
5. **Clean comments**: label major SVG sections (chassis, panels, ports, LEDs)
