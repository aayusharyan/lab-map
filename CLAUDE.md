## Data Files & Directory Structure

### Overview

```
config/            ← User-editable configurations (committed to git)
  ├── physical.json
  ├── traffic.json
  ├── vlan.json
  └── settings.json

public/data/       ← Runtime data (gitignored, generated by watcher)
  └── *.json       ← App loads data from here at runtime

schema/            ← JSON schemas for validation
  └── *.schema.json
```

### Data Files

| File              | Purpose                                                     | Schema Location               |
| ----------------- | ----------------------------------------------------------- | ----------------------------- |
| **physical.json** | Physical network topology (nodes, connections, coordinates) | `schema/physical.schema.json` |
| **traffic.json**  | Traffic flow data between nodes                             | `schema/traffic.schema.json`  |
| **vlan.json**     | VLAN configuration and assignments                          | `schema/vlan.schema.json`     |
| **settings.json** | Application settings and UI preferences                     | `schema/settings.schema.json` |

### Config Directory

- **Location**: `config/` (root of repository)
- **Purpose**: User-editable configurations committed to git; Docker image ships these so visualizations work out of the box
- **Git status**: Committed to version control
- **Docker**: Copied into image via `COPY config ./config`; a volume mount at runtime overrides these defaults

### Runtime Data Directory

`public/data/` is **gitignored** — it is generated by the watcher on startup and must not be committed.

### Validation

- **Tool**: `ajv` validates data files against JSON schemas in `schema/`
- **Scripts**:
  - `pnpm dev` - Runs Vite dev server + auto-validation watcher
  - `pnpm watch` - Standalone watcher (auto-validate on file changes)
- **File**: [watcher.js](watcher.js)

### Design Pattern

- **Flat colors only** - no gradients, use solid fill colors
- **Consistent stroke widths**: `0.7` (details), `0.8` (panels), `1.2-1.5` (chassis outline)
- **Rounded corners**: `rx="1"` for small elements, `rx="2"` or `rx="3"` for larger panels
- **Simple shapes**: rectangles, circles, lines - avoid complex paths where possible
- **Clean SVG comments**: label major sections (chassis, panels, ports, LEDs)
- **Both variants required**: `dark.svg` and `light.svg` for each icon

## Code Documentation Style

Document files in FOSS style with these principles:

### Structure

1. **File purpose** - Start with what the file does and why it exists (1-2 lines)
2. **Usage examples** - Show how to use it (commands, imports, etc.) if applicable
3. **Step-by-step** - Explain each major section, function, or code block
4. **Customization notes** - Highlight what users can modify

### Avoid

- **Redundant project info** - Don't repeat project name, author, or license in every file (it's already in package.json/root)
- **Obvious comments** - Skip comments like "import React" or "define variable"
- **Over-documentation** - Only document what isn't self-evident from the code

### Format

- **TypeScript/JavaScript**: Always use multi-line comment syntax (`/* */` or `/** */`) - never use single-line `//`
- **Other languages**: Use language-appropriate style (e.g., `#` for shell)
- Section separators: Use consistent dividers for major sections
- Keep comments concise - prefer one-liners when possible

### GitHub Actions Workflows (.github/workflows/\*.yml)

Follow the same shell-style header pattern as Docker/Shell files:

```yaml
# =============================================================================
# @file filename.yml
# @description Brief description
#
# Detailed usage instructions and notes.
#
# Usage:
#   How to trigger the workflow
#
# Flow:
#   Step1 → Step2 → Step3
# =============================================================================

name: Workflow Name
```

Section separators within workflow (same as Dockerfile):

```yaml
# ---------------------------------------------------------------------------
# Section Name
# ---------------------------------------------------------------------------
```

### GitHub Templates (.github/ISSUE_TEMPLATE, .github/PULL_REQUEST_TEMPLATE)

- **No file headers** - these are user-facing templates
- Use HTML comments for internal instructions: `<!-- explanation -->`
- Keep templates clean and focused on user input

### Required Coverage

- **Document EVERY major code block** - not just file headers
- Add comments above: interfaces, types, functions, constants, class definitions
- Each section must have its own comment explaining what it does
